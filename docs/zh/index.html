AAMS / 1.0

每个代理。
一个文件。

永远不再从零开始。 AAMS 是一个声明式标准，为 AI 代理提供工作区、记忆和规则集 —— 跨越每个工具、每次会话、每个仓库。

实际效果

40 秒。一个文件。无需设置。你的 AI 代理记住一切 —— 跨会话、跨工具。这就是 AAMS：你仓库的食谱。

问题

你克隆一个仓库。你的代理从零开始。没有上下文，没有记忆， 没有约定的结构。每次会话都重新发明轮子。 每个工具（Copilot、Cursor、Claude、Codex）都需要同样的解释。

AAMS 用一个文件解决这个问题。

  • 一个清单 — 仓库根目录的 .agent.json。每个代理都会读取。每个工具都会读取。
  • 一个工作区 — WORKING/ 结构，包含工作文件、白皮书、日记、记忆。
  • 一个契约 — 代理可以做什么，不能做什么，从哪里开始。

10 秒开始

bash — 你需要的唯一文件
curl -sO https://raw.githubusercontent.com/DEVmatrose/AAMS/main/.agent.json
  1. 1
    将 .agent.json 放入仓库 一条 curl 命令。仅此而已。
  2. 2
    告诉代理：开始。 "Read .agent.json and execute agent_contract.on_first_entry."

接下来会发生什么

代理自动引导自身：

  • 创建 WORKING/ — 工作文件、白皮书、日记、记忆、指南、日志
  • 扫描仓库 — 分析代码库，检测架构和依赖关系
  • 撰写第一份工作文件 — 记录发现的内容和下一步计划
  • 索引到 LTM — 构建跨会话的长期记忆

可验证。每一步都是仓库中的 Markdown 文件。git log 显示历史。ltm-index.md 显示代理当时的认知。不再有"谁改了这里，为什么？"—— 你直接 grep 工作文件存档。

下一次会话？代理读取记忆，创建新的工作文件， 从上次停下的地方继续。没有工具锁定。 明天从 Cursor 切换到 Claude Code —— 上下文不变。

适用于每个代理

Cursor 有 .cursorrules。 Copilot 有 copilot-instructions.md。 Claude 有 CLAUDE.md。 每个工具都有自己的约定。AAMS 用一个契约替代它们。

Copilot Cursor Claude Code Codex Windsurf Aider Continue.dev Firebase Studio

不需要 CLAUDE.md。不需要 GEMINI.md。不需要 airules.md。

"我不需要这个。我自己管理上下文。"

想象你奶奶的食谱 —— 充满了精彩的配方、手写笔记 和几十年积累的智慧。但没有目录。没有交叉引用。 找不到你需要的东西，除非翻遍整本书。

AAMS 是有体系的食谱。你奶奶的食谱所缺少的纪律。 目录、搜索、版本历史 —— 为你的仓库而设。

5 次会话你还记得一切。50 次会话你重新决定两个月前已经决定的事情。 100 次会话你的代理开始为已经解决的问题幻觉出解决方案。

你不是在增加开销。你是在消除你不知道自己有的开销。

AAMS 不是框架

AAMS 不是工具。不是运行时。不是需要安装的框架。 AAMS 是一个结构化的上下文和决策编译器。

人类定义结构、规则和上下文。代理是确定性的工作者 —— 不是行动者。 记忆不是代理"拥有"的东西 —— 记忆是聚合的、可追溯的历史。

核心循环：

输出 → 文档 → 决策 → 记忆 → 新上下文

开始使用

一个文件。就这么简单。

bash — 你需要的唯一文件
curl -sO https://raw.githubusercontent.com/DEVmatrose/AAMS/main/.agent.json

将 .agent.json 放入仓库根目录。用 reference/prompts/bootstrap.md 中的引导提示交给代理。完成。

想深入了解或构建框架？

  • AGENT.json — 完整注释的参考清单，包含所有字段和选项。 面向框架构建者和实现者。
  • SPEC.md — 完整规范。1,070+ 行。 每个字段、每条规则、每个模式的规范定义。
  • WORKING/ — 本仓库自己的实时工作区。标准 应用于自身。每次会话都有文档记录、索引、可追溯。
